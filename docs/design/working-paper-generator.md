# Working Paper Generator

**Module:** `spectrum_systems/modules/working_paper_generator.py`  
**Contract Schema:** `contracts/schemas/working_paper.schema.json`  
**Example:** `contracts/examples/working_paper.json`  
**Tests:** `tests/test_working_paper_generator_module.py`

---

## Philosophy: Clearest Argument Wins

The working paper generator is designed around one principle: **the clearest argument
wins, not the loudest voice**.

In spectrum coordination meetings, dominant voices, strong personalities, and
presentation order all introduce bias into what gets recorded. The generator
counteracts this by:

- Deriving findings **from decisions and technical claims**, not from speaker
  prominence or repetition.
- Attaching **traceability** to every claim so reviewers can verify the source.
- Surfacing **gaps and unresolved issues** that were discussed but not resolved.
- Generating **calibrated agency questions** that expose assumptions rather than
  accept them.

The output is defensible before FCC and NTIA because every assertion traces back
to a source (transcript segment, slide claim, or gap analysis finding) with an
associated confidence score.

## Section Definitions

### `title`
Derived from the `meeting_title` or `title` field of the structured transcript.
Falls back to a generic working-paper title.

### `executive_summary`
5–8 sentences. Covers:
- What was studied (band range if detected)
- Key tension (spectrum sharing vs. incumbent protection)
- 1–2 main findings from decisions or slide claims
- At least one unresolved issue

### `purpose_and_scope`
States the purpose of the paper, the band range(s) under analysis, and whether
the scope is in-band, adjacent-band, or both. Enriched by slide metadata when
available.

### `system_description`
Summarizes federal systems (e.g., P2P links, radar) and non-federal systems
(e.g., CBRS base stations, 5G NR) drawn from:
- Named entities in the structured transcript
- Entities extracted from slide intelligence
- Keyword-based fallbacks from raw transcript text

### `technical_analysis`
Organized into three subsections:
- **Assumptions** — from explicit assumption fields or keyword extraction
- **Methods** — from link budget, propagation, and modeling references
- **Limitations** — from sentences containing uncertainty/gap language

When gap analysis is provided, two additional subsections are injected:
- **Unaddressed Deployment Assumptions** — from `deployment_gaps`
- **Unvalidated Interference Scenarios** — from `interference_gaps`

### `key_findings`
Atomic bullet findings derived from:
1. `decisions_made` in the structured transcript
2. Technical sentences about interference or protection that do not contain uncertainty language
3. High-confidence claims from slide intelligence (confidence ≥ 0.7)

Maximum 10 findings. Deduplicated by normalized text prefix.

### `risks_and_uncertainties`
Compiled from:
1. `risks_or_open_questions` in the structured transcript
2. Gap analysis `gaps` (prefixed with `[Gap]` for identification)
3. Fallback structured risks if none are found (model uncertainty, deployment assumptions, coordination challenges)

### `decisions_and_recommendations`
Normalized from:
1. `decisions_made` in the structured transcript
2. `action_items` in the structured transcript
3. Implied recommendations if neither is present

### `open_questions_for_agencies`
Calibrated questions generated by `generate_agency_questions()`. Always at least
5 questions. Framed in one of four patterns:
- `"How would [agency] validate [finding]?"`
- `"What assumptions support [risk]?"`
- `"What would it take to implement [recommendation] without causing interference?"`
- `"What additional data is needed to address [gap]?"`

### `appendix.source_traceability`
Every claim added to the paper is recorded here with:
- `claim` — the claim text (truncated to 120 chars)
- `source` — one of `transcript | slide | gap_analysis | derived`
- `confidence` — float in [0, 1]

This appendix is **non-negotiable** for FCC/NTIA defensibility.

### `appendix.slide_alignment_summary`
Maps slide claims to working paper sections. Only populated when slide signals
are provided. Each entry: `slide_claim`, `section`, `confidence`.

### `appendix.discussion_gaps`
Slide vs. discussion conflicts from the gap analysis `conflicts` field.

## Data Flow

```
transcript + slides (optional) + gap_analysis (optional)
        │
        ▼
generate_working_paper(structured_transcript, slide_signals, gap_analysis)
        │
        ├─► _generate_executive_summary()   ──► executive_summary
        ├─► _generate_purpose_and_scope()   ──► purpose_and_scope
        ├─► _generate_system_description()  ──► system_description
        ├─► _generate_technical_analysis()  ──► technical_analysis
        ├─► _generate_key_findings()        ──► key_findings
        ├─► _generate_risks_and_uncertainties() ──► risks_and_uncertainties
        ├─► _generate_decisions_and_recommendations() ──► decisions_and_recommendations
        ├─► _generate_gap_augmented_sections() ──► injected into technical_analysis
        ├─► _build_slide_alignment_summary() ──► appendix.slide_alignment_summary
        │
        └─► generate_agency_questions(working_paper, gap_analysis)
                ──► open_questions_for_agencies
```

All traceability entries are accumulated in a shared `traceability` list during
generation and written to `appendix.source_traceability` at the end.

## Pipeline Integration

The working paper generator is the final stage in the meeting-minutes pipeline:

```
transcript_text
    └─► stage_load
        └─► stage_extract(structured_extraction)
            └─► stage_signals(signals)
                └─► stage_build_study_state
                    └─► stage_package
                        └─► [optional] build_slide_intelligence_packet(slide_deck)
                            └─► generate_working_paper(structured_extraction,
                                                        slide_signals, gap_analysis)
                                └─► package_result["working_paper"]
```

The working paper stage always runs. Missing optional inputs produce valid (if
minimal) output. The pipeline never fails due to the working paper stage.

## Design Constraints

| Constraint | Implementation |
|---|---|
| Deterministic | No LLM calls, no network I/O, no randomness |
| Module-first | Lives entirely within `spectrum_systems/modules/` |
| Backward compatible | All inputs except `structured_transcript` are optional |
| No breaking changes | Pipeline `run_pipeline()` signature unchanged except `gap_analysis` added |
| Traceability | Every claim written to `appendix.source_traceability` |
| Validation | `validate_working_paper()` enforces required keys and types |
