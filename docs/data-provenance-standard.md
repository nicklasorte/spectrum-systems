# Data Provenance Standard

## Overview
This document defines the provenance standard for all data, artifacts, and AI-assisted outputs described in the Spectrum Systems Lab Notebook.

The standard exists to support:
- traceability
- reproducibility
- reviewability
- version control
- confidence assessment
- trustworthy reuse

For how to rerun workflows identically, see `docs/reproducibility-standard.md`; provenance focuses on lineage and accountability, while reproducibility focuses on repeatable execution.

Provenance must answer these questions for every important artifact:
- What is this?
- Where did it come from?
- Who created or transformed it?
- When was it created or modified?
- What source inputs were used?
- What workflow or system generated it?
- What version is it?
- Has a human reviewed it?
- How confident are we in it?

## Design Basis
The standard is based on a practical combination of:
- entity / activity / agent thinking
- derivation and version tracking
- rich metadata for reuse
- persistent identifiers and source linkage
- human review and AI traceability

## Core Provenance Model
### Entity
A data object, document, table, figure, transcript extract, schema record, prompt output, report paragraph, or any other artifact.

### Activity
A process that creates, transforms, extracts, normalizes, summarizes, validates, or reviews an entity.

### Agent
A human, team, workflow, script, model, or AI system responsible for creating, modifying, reviewing, or approving an entity.

## Core Relationships
- entity was generated by activity
- activity used entity
- activity was associated with agent
- entity was derived from entity
- entity was reviewed by agent
- entity belongs to version lineage

## Required Provenance Fields
Every important schema or structured artifact should support these fields unless clearly not applicable.

- record_id: stable identifier for the record.
- record_type: classification of the record or artifact type.
- title: short human-readable name.
- description: concise explanation of the artifact.
- source_document: name or identifier of the originating document.
- source_document_id: canonical ID for the source document.
- source_location: section, page, timecode, or coordinate in the source.
- source_url_or_path: link or path to where the source can be retrieved.
- source_revision: version or revision tag of the source.
- timestamp_created: when the record was first created.
- timestamp_modified: when the record was last changed.
- created_by: agent identifier that created the record.
- modified_by: agent identifier that last modified the record.
- generated_by_system: workflow, tool, or model that produced the record.
- workflow_name: workflow that generated or transformed the record.
- workflow_step: step within the workflow responsible for the change.
- derived_from: identifiers of parent entities the record depends on.
- related_entities: other linked entities that provide context.
- version_id: unique identifier for this version instance.
- version_number: ordered version number within the lineage.
- review_status: status of human review.
- reviewed_by: agent identifier for the reviewer.
- review_date: timestamp of the review action.
- confidence_score: confidence level for the content.
- quality_status: processing or curation state of the record.
- usage_constraints: limits, caveats, or handling instructions.
- notes: freeform additional context.

## Standard Enumerations
review_status:
- unreviewed
- pending_review
- reviewed
- approved
- rejected

quality_status:
- raw
- extracted
- normalized
- generated
- validated
- superseded

confidence_score:
- low
- medium
- high

agent_type:
- human
- ai_model
- script
- workflow
- team

record_lineage_type:
- original
- derived
- merged
- corrected
- superseded

Schemas may extend these values, but should not redefine them casually.

## Provenance Rules
1. Every structured record must have a stable identifier.
2. Every generated record must indicate its source document or parent entities.
3. Every AI-generated output must record the generating system or workflow.
4. Every AI-generated output that matters downstream must have a review status.
5. Every change that affects interpretation must increment version information.
6. Superseded records should not be deleted; they should be marked as superseded.
7. Source metadata should persist even if the underlying source file later disappears.
8. Confidence scores must never replace human review.
9. Derived artifacts should preserve references to all critical parent artifacts.
10. If provenance is unknown, it should be explicitly marked unknown rather than omitted.

## Provenance Lifecycle
Source Collected -> Entity Created -> Activity Performed -> Derived Artifact Generated -> Human Review -> Approved / Rejected / Superseded -> Stored in System Memory

- Source Collected: identify and register the originating document or dataset with revision markers.
- Entity Created: create the first structured representation with identifiers and minimal context.
- Activity Performed: run transformations, normalizations, or extractions with workflow and agent attribution.
- Derived Artifact Generated: produce secondary artifacts with explicit derivation links.
- Human Review: capture reviewer, date, and review status for outputs that influence downstream steps.
- Approved / Rejected / Superseded: record disposition and update version lineage without deleting history.
- Stored in System Memory: persist entities, relationships, and metadata so they remain usable even if sources move.

## Worked Examples
### Example 1 — Comment Resolution Record
- source: Draft Spectrum Sharing Report v1.1 Section 3.2
- generating activity: disposition drafting workflow step 4 (response synthesis)
- responsible agent: Comment Resolution Engine assisted by Analyst A
- version: v1.2 with version_id CMT-2025-014-v1.2
- review_status: approved by Reviewer One on 2025-02-18
- confidence_score: high
- derived_from: source comment CMT-2025-014 and source_document DOC-2025-022

### Example 2 — Transcript Extraction Record
- source: TIG meeting transcript MTG-2025-041 timecode 00:12:15
- generating activity: transcript-to-issue extraction workflow step 2 (entity extraction)
- responsible agent: Transcript-to-Issue Engine with Analyst B validating
- version: v1.0 with version_id TRN-2025-041-ISSUE-01
- review_status: pending_review by Reviewer Two
- confidence_score: medium
- derived_from: transcript MTG-2025-041 and meeting metadata

### Example 3 — Study Artifact Record
- source: simulation outputs RUN-2025-118 and assumption set ASM-2025-007
- generating activity: study artifact generator workflow step 5 (table rendering)
- responsible agent: Study Artifact Generator v0.3 with Analyst C
- version: v1.1 with version_id ART-2025-031-v1.1
- review_status: reviewed by Reviewer Three on 2025-03-02
- confidence_score: high
- derived_from: simulation run RUN-2025-118, assumptions ASM-2025-007 and ASM-2025-003
