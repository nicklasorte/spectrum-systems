# Governed Prompt Queue Next-Step Orchestration — Codex Surgical Review

## Scope reviewed
- `spectrum_systems/modules/prompt_queue/next_step_orchestrator.py`
- `spectrum_systems/modules/prompt_queue/next_step_queue_integration.py`
- `contracts/schemas/prompt_queue_next_step_action.schema.json`
- minimal state-machine context required for this slice (`spectrum_systems/modules/prompt_queue/queue_state_machine.py`)

## Decision
FAIL

## Critical Findings (max 5)

1. **Semantic mismatch between `decision_status` and `action_status` is not rejected during queue integration.**
   - **What is wrong:** `apply_next_step_action_to_queue()` derives target state from `decision_status`, but spawn/no-spawn behavior from `action_status`, without enforcing that the pair is valid for the slice contract.
   - **Why dangerous:** A schema-valid but semantically inconsistent artifact can produce the wrong next action, create duplicate/unintended children, and advance the loop unsafely.
   - **Location:** `spectrum_systems/modules/prompt_queue/next_step_queue_integration.py` (`_decision_to_target_status`, `apply_next_step_action_to_queue`).
   - **Realistic failure scenario:** An artifact with `decision_status="complete"` and `action_status="spawn_review"` passes schema validation, transitions the work item to `complete`, and still spawns a review child—contradicting governance intent and creating unintended loop continuation.

2. **`action_reason_code` is validated only by enum, not by deterministic pairing to the selected decision/action path.**
   - **What is wrong:** Neither schema nor integration enforces that reason code corresponds to the exact `(decision_status, action_status)` mapping.
   - **Why dangerous:** Audit trail can claim one rationale while system executes another, obscuring wrong-action or unsafe-advance incidents.
   - **Location:** `contracts/schemas/prompt_queue_next_step_action.schema.json` and `spectrum_systems/modules/prompt_queue/next_step_queue_integration.py`.
   - **Realistic failure scenario:** Artifact states `decision_status="reentry_blocked"`, `action_status="blocked_no_action"`, but uses a review/reentry reason code; downstream governance/evidence review misattributes why no child was (or was not) created.

## Required Fixes

1. **Enforce a single canonical decision→action tuple during integration (fail closed).**
   - In `apply_next_step_action_to_queue()`, verify the incoming artifact matches the canonical tuple for the given `decision_status` (both `action_status` and `action_reason_code`). Reject any mismatch before state transition.

2. **Encode deterministic pairing constraints in the action schema.**
   - Add `allOf` + `if/then` clauses in `prompt_queue_next_step_action.schema.json` so each `decision_status` permits exactly one `action_status` and one `action_reason_code`.

## Optional Improvements

1. **Add explicit negative tests for semantically inconsistent but enum-valid artifacts.**
   - Include cases such as `complete + spawn_review`, `review_required + marked_complete`, and reason-code mismatches to prove fail-closed behavior.

## Trust Assessment
NO. In current form, the slice is deterministic when artifacts are internally generated by the orchestrator, but it does not fail closed against semantically inconsistent-yet-schema-valid action artifacts at integration time. That leaves a realistic path to wrong action selection, unintended child creation, and unsafe loop advancement.

## Failure Mode Summary
Worst realistic failure: a malformed/tampered next-step action artifact with mixed semantics (e.g., terminal decision + spawn action) is accepted by integration, causing a terminal work item to be marked complete while still spawning a child. This simultaneously corrupts lineage intent, creates unintended duplicate/extra loop work, and undermines governance confidence in queue progression.
