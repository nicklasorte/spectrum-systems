# Spectrum Systems — PQX Queue Roadmap
**Date:** 2026-03-28

## INTENT
Define a repo-native, fail-closed roadmap for turning the existing governed prompt queue seams in `spectrum-systems` into a deterministic PQX Queue layer that can execute ordered work items, classify outcomes, advance only on explicit control decisions, preserve replayability, and certify queue completion without inventing a second control system.

## System Goal
Build an artifact-first PQX Queue that runs one governed step at a time through the existing eval → control → enforcement architecture, records queue state and decision artifacts at every transition, halts on ambiguity or missing evidence, supports deterministic replay/resume, and only declares queue completion after certification and audit-bundle emission.

## Architectural Invariants
- The queue is an orchestration layer only; it must never embed business logic that belongs in eval, control, enforcement, or certification modules.
- No queue step may advance without an explicit, schema-valid decision artifact.
- Queue state is authoritative and must be artifact-backed, replayable, and checkpointed.
- Missing evidence, ambiguous reports, invalid schemas, or uncertain attribution are fail-closed conditions.
- Retry, blocked recovery, and remediation are governed branches, never silent behavior.
- Queue completion is not equivalent to step exhaustion; completion requires certification.
- Every queue mutation must preserve artifact lineage, trace continuity, and audit traceability.
- Existing prompt queue seams are the primary implementation surface; add new queue components only where the repo has no current seam.
- Multi-queue scheduling is out of scope until single-queue governance is complete.

## Execution Rules (PQX)
- Each row is one atomic implementation slice.
- Prefer MODIFY EXISTING over ADD NEW.
- No unguided invention: roadmap steps must bind to repo seams, existing prompt queue artifacts, or explicit SOURCE GAP (FILLED).
- Queue steps must define input artifact refs, output artifact schemas, success criteria, and failure classification paths.
- Queue progression must follow dependency order; no downstream queue automation before manifest, state, parsing, transition, and certification seams exist.
- All PQX queue slices must include tests, artifact outputs, and fail-closed guarantees.
- Queue must never advance on free-text judgment alone.
- No retry without a recorded retry or remediation decision artifact.
- No queue completion without certification and audit-bundle emission.
- Serial safety is mandatory before any later queue-scale automation.

## FAILURE MODES PREVENTED
- Silent queue advancement on malformed or ambiguous step reports.
- Fragmented next-step logic spread across queue modules without a single governed decision spine.
- Queue progress that cannot be replayed or resumed from the correct point.
- Retry or blocked recovery behavior that mutates queue state without explicit governance artifacts.
- Queue completion being inferred from status exhaustion rather than certification.
- Audit trails that fragment across work-item artifacts without a queue-level evidence package.
- Policy drift in queue transition/retry behavior without replay-backed backtesting.
- Premature scheduler expansion before single-queue trust boundaries are closed.

## GUARANTEES PROVIDED
- Queue execution remains artifact-first and fail-closed.
- Every queue phase can be mapped to existing repo seams or explicit, bounded SOURCE GAP (FILLED) work.
- Mandatory queue components are all present in the roadmap.
- Existing governed prompt queue work is treated as implemented but incomplete, not replaced.
- The roadmap preserves eval → control → enforcement integrity and adds queue-specific certification/auditability only after state, transition, replay, and observability foundations exist.
- Each row is PQX-expandable and small enough to implement as a governed slice.

## KNOWN LIMITS
- The repository already contains substantial governed prompt queue implementation, but it does not yet expose a single manifest-driven queue runner, queue certification gate, queue audit bundle, or policy-backtested queue governance loop.
- Current queue slices are strong at per-step governance, but transition logic is still distributed across multiple policy modules and reports.
- Multi-queue scheduling remains explicitly out of scope.
- This roadmap does not claim production scheduler readiness; it reconstructs the minimum dependency-valid path from the current repo state.

## NEXT RISKS
- Leaving queue manifest/state ununified will keep execution dependent on manual prompt ordering.
- Failing to consolidate transition policy will preserve hidden advancement paths across post-execution, next-step, retry, and recovery modules.
- Adding certification before replay/resume or audit-bundle support would create a false “done” signal.
- Attempting multi-queue orchestration before queue policy backtesting would reintroduce ambiguity and attribution risk.
- If queue observability is not normalized against the execution loop, health signals will remain advisory instead of governable.
